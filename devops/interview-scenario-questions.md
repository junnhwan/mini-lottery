# 抽奖项目面试场景题清单

## 高并发与性能

- 高并发抽奖瞬时打满时，系统最先可能被哪一层打爆？你会怎么保护？
- 如果同一时间有 10 万请求打到抽奖接口，你会如何定位瓶颈在应用、Redis、MySQL 还是 MQ？
- 为什么你的库存扣减不直接走 MySQL？在什么并发量级下会出现明显差异？
- 如果 Redis QPS 已经很高了，你还会怎么继续优化抽奖主链路？
- 抽奖算法为什么要做预装配，而不是每次实时算概率？
- O(1) 方案在什么情况下会反而不划算？
- 如果奖品数量增加到几百个，当前抽奖算法还适合吗？

## 库存与防超卖

- 活动库存和奖品库存分别解决什么问题？为什么要分两层库存？
- 如果活动库存还够，但某个奖品库存没了，系统应该如何处理？
- 如果 Redis 扣库存成功了，但后续订单写库失败了，怎么补偿？
- 为什么当前方案要用 Lua 脚本把 `DECR + SETNX` 合成一次原子执行？
- 如果运营手动回补库存，当前库存方案会有什么风险？
- 如果 `SETNX` 冲突了，为什么当前实现直接失败而不是发兜底奖品？
- 如果改成 `INCR` 已售计数模型，会比当前方案更好吗？为什么？
- 如果 Redis 和 DB 库存最终不一致，你怎么做对账和修复？

## 缓存与一致性

- 如果 Redis 宕机，抽奖服务应该直接报错、降级到 DB，还是限流后部分服务？为什么？
- 如果本地缓存和 Redis 缓存数据不一致，你怎么判断哪个为准？
- 规则配置改了，但服务还在读旧缓存，怎么快速生效？
- 多实例部署下，本地缓存不一致怎么解决？
- 为什么缓存预热要做在启动阶段或装配阶段，而不是等第一次请求再加载？
- 如果缓存雪崩、击穿、穿透同时出现，你这个项目最可能先暴露哪种问题？

## MQ 与异步解耦

- 为什么发奖和库存落库要异步，而不是放在一个本地事务里同步完成？
- `afterCommit` 能解决什么问题？它解决不了什么问题？
- 如果事务提交成功后，服务在发送 MQ 前挂了，消息怎么办？
- 如果生产者发送失败、Broker 崩溃、消费者挂掉，各自该怎么处理？
- 为什么说 `afterCommit` 不等于可靠消息最终解决方案？
- 你的消费者如何保证幂等？哪些链路还不够严格？
- 为什么库存落库现在只能算“最终一致”，不是“强一致”？
- 如果 RabbitMQ 整体不可用，抽奖主流程还能不能继续？如何降级？

## 规则引擎与业务设计

- 为什么责任链放前面、规则树放后面？反过来行不行？
- 为什么黑名单规则放责任链，而次数锁和库存判断放规则树？
- 权重规则为什么命中后直接返回，不继续往下走默认链？
- 权重子奖池是如何缩小奖池范围的？
- 如果面试官质疑“规则配置化增加复杂度”，你怎么解释这个取舍？
- 哪些规则适合配置化，哪些规则更适合写死在代码里？
- 当前规则热更新真正做到哪一步了？还有哪些缺口？

## 限流与风控

- 为什么对超频用户返回兜底奖品，而不是直接返回 `429`？
- 令牌桶为什么适合这个抽奖接口，而不是滑动窗口？
- 当前限流是单机的还是分布式的？多实例下会有什么问题？
- 拉黑策略如果误伤正常用户，你怎么设计恢复机制？
- 限流和黑名单在风控上分别解决什么问题？

## 故障恢复与补偿

- 如果应用在扣完 Redis 库存后立刻宕机，会留下什么半成功状态？
- 如果延迟队列没有成功触发，超时订单没有回滚怎么办？
- 如果发奖任务执行到一半失败，如何避免重复发奖或漏发奖？
- 你会如何设计定时补偿任务来兜底 MQ、库存、订单状态不一致的问题？
- 生产环境里你会重点监控哪些指标来发现抽奖链路异常？

## 架构取舍

- 为什么不用分布式锁串行扣减，而选择现在这套 Lua + MQ + 定时补偿方案？
- 为什么不用 Kafka，而用 RabbitMQ？
- 为什么不用定时任务扫超时订单，而是用 TTL + 死信队列？
- 为什么不用 ZSet 做概率抽奖？
- 为什么不用数据库事务把所有事情一次做完？
