# 简历项目针对性问题清单

## 一、项目整体与目标

- 这个抽奖平台的核心业务目标是什么？
- 你在这个项目里最想解决的 3 个核心问题分别是什么？
- 如果让你用一句话总结项目亮点，你会怎么说？
- 这个项目最难的技术点和最难的业务点分别是什么？
- 如果让你重新做一遍，这个项目你最想重构哪一块？

## 二、责任链与规则树

- 为什么前置用责任链，后置用规则树？
- 责任链主要解决什么问题？规则树主要解决什么问题？
- 黑名单、权重、默认规则为什么适合放责任链？
- 次数锁、库存扣减、兜底奖品为什么更适合放规则树？
- 责任链节点为什么要用 `@Scope(PROTOTYPE)`？
- 责任链缓存为什么要按 `activityId` 维度缓存？
- 规则树的节点、边、跳转条件是怎么设计的？
- `ALLOW` 和 `TAKE_OVER` 的语义是什么？
- 如果规则配置写错了，比如链路断了、树里没有出口，会发生什么？
- 当前规则热更新是真的实现了吗？还是只停留在设计层？

## 三、抽奖算法设计

- O(1) 抽奖算法的核心思想是什么？
- O(log n) 抽奖算法的核心思想是什么？
- 为什么概率装配到 Redis Hash 就能做到 O(1)？
- 为什么大范围概率时要切换到区间表 + 二分？
- `rateRange` 是怎么计算的？为什么要 `stripTrailingZeros`？
- O(1) 方案为什么要 `shuffle`？不 `shuffle` 会怎样？
- O(log n) 方案为什么不能 `shuffle`？
- 两种算法的切换阈值为什么设成 `10000`？
- 这两种方案的空间复杂度和时间复杂度分别是什么？
- 如果面试官说“你这里概率装配会有精度误差”，你怎么回答？

## 四、库存扣减与一致性

- 你的项目里为什么要区分活动库存和奖品库存？
- 活动库存和奖品库存分别在什么时机扣减？
- 为什么活动库存先扣，奖品库存后扣？
- 为什么要把 `DECR + SETNX` 放进 Lua 脚本里一次执行？
- 为什么不用纯分布式锁方案？
- 为什么 Lua 脚本里还要保留 `SETNX` 分段锁？
- `SETNX` 分段锁防的到底是什么场景？
- 如果 `SETNX` 冲突，当前代码为什么直接失败？
- 如果 Redis Lua 扣减成功但 DB 没同步成功，会怎样？
- DB 侧 `WHERE remain_stock > 0` 解决了什么问题，没解决什么问题？

## 五、RabbitMQ 与异步处理

- 为什么发奖要异步处理？
- 为什么库存落库也要异步处理？
- 为什么消息是在 `afterCommit` 里发送，而不是事务中发送？
- `afterCommit` 和 `afterCompletion` 分别解决什么问题？
- 当前 MQ 可靠性方案做到什么程度？还有哪些缺口？
- 生产者发送失败、消息丢失、重复消费分别怎么处理？
- 发奖消费者是如何做幂等的？
- 当前库存落库消费者为什么说不算严格幂等？定时补偿又兜底了什么？
- 延迟队列为什么用 `TTL + 死信队列` 实现，而不是定时任务扫单？
- 如果 RabbitMQ 不可用，你会怎么降级？

## 六、限流与风控

- 为什么选 Guava `RateLimiter`，不用滑动窗口？
- 令牌桶在这个抽奖场景里的优势是什么？
- 为什么超频用户要拉黑，而不是只做限流？
- 为什么限流后返回兜底奖品，而不是直接报错？
- 当前限流方案是单机还是分布式？
- 多实例部署时，当前限流实现会有什么问题？
- 如果要做更严格的风控，你还会补哪些策略？

## 七、配置化与扩展性

- 为什么把规则放到 DB，而不是写死在代码里？
- 规则配置化给项目带来了哪些复杂度？
- 你如何平衡“可配置”与“可维护”？
- 如果运营频繁改规则，当前系统支持到什么程度？
- 当前项目的热更新能力存在哪些不完整点？

## 八、项目缺陷与改进

- 你认为这个项目当前最明显的三个缺陷是什么？
- 如果把它往生产级系统演进，你优先改哪三点？
- 你会怎么补全 MQ 可靠消息能力？
- 你会怎么补全缓存失效和规则热更新能力？
- 你会怎么把当前库存方案改得更稳健？
- 你会怎么加强监控、告警和对账能力？
