## 机器情况

阿里云 2c2g 云服务器，docker部署，mysql、redis、rabbitmq、java-app



## 场景一

数据非常不错！来解读一下：

| 指标           | 数值           | 含义                                   |
| -------------- | -------------- | -------------------------------------- |
| **Throughput** | **1209.8/sec** | QPS ≈ 1210，单机每秒处理 1210 次抽奖   |
| Average        | 20ms           | 平均响应时间，很快                     |
| Median         | 16ms           | 50% 的请求在 16ms 内完成               |
| 90% Line       | 28ms           | P90 = 28ms，优秀                       |
| 99% Line       | 86ms           | P99 = 86ms，良好                       |
| Max            | 525ms          | 最大延迟，可能是首批连接建立或 GC 停顿 |
| Error %        | 0.00%          | 零错误，系统稳定                       |
| # Samples      | 10,000         | 50 并发 × 200 次 = 10000 请求全部完成  |

**总结：单机 QPS ~1200，P99 < 100ms，零错误率。** 这对于一台小规格云服务器（走公网到 Redis/MySQL）来说是很好的数据。

**面试可以这样说：**

> "50 并发压测 1 万次请求，单机 QPS 约 1200，P99 延迟 86ms，零错误率。瓶颈主要在规则链路、数据库写入和网络 RTT。如果继续优化，可以进一步做消息可靠投递、批量落库或者多实例水平扩展。"

## 场景3

场景 3 结果分析：

| 指标    | 数值          | 说明                                     |
| ------- | ------------- | ---------------------------------------- |
| QPS     | **104.2/sec** | 比基准(1210)低很多，因为库存扣减路径更重 |
| Average | 837ms         | 包含责任链→规则树→Lua 扣库存→写DB→发MQ   |
| P99     | 2836ms        | 高并发竞争同一库存，尾部延迟大           |
| Error % | 0.00%         | HTTP 都是 200，库存不足的是业务码拒绝    |

QPS 从 1210 降到 104 是正常的 —— 场景 1 用的是压测活动（轻路径），场景 3 走了完整的库存扣减链路（责任链 + 规则树 + Redis Lua 原子扣减 + 写订单 + 发 MQ），开销大很多。

**关键验证：有没有超卖？** 去云服务器 MySQL 查一下：



```sql
-- 1. 剩余库存
SELECT activity_id, total_stock, remain_stock FROM activity WHERE activity_id = 'A20260310001';

-- 2. 成功订单数
SELECT COUNT(*) AS success_orders FROM lottery_order WHERE activity_id = 'A20260310001' AND status IN (0, 1);

-- 两者之和应 = 1000（初始库存）
```

如果 `成功数 + 剩余库存 = 1000`，就证明 **Redis + Lua 扣库存方案不超卖**，这是面试最有说服力的数据。
